# Coding Guidelines

This document outlines the coding standards and best practices for the board game library project. It serves as a guide for both developers and LLMs to ensure consistency and maintainability across the stack.

## 1. Backend (Go, Gin, oapi-codegen, sqlc)

The backend is built with Go using the Gin web framework. It follows a contract-first approach using OpenAPI.

### Architecture
- **Contract-First**: The `swagger/api.yaml` is the source of truth for the API.
- **Generated Code**:
    - API models, server interface, and client are generated by `oapi-codegen` into `backend/api/oapi-codegen.gen.go`.
    - Database access code is generated by `sqlc` into `backend/db/`.
- **Handwritten Code**:
    - API handlers (implementing `ServerInterface`) in `backend/api/`.
    - Business logic and validation utilities in `backend/api/utils.go`.
    - Database configuration and connection management in `backend/db/database.go`.
    - Integration tests in `backend/tests/`.

### Guidelines
- **Use Modern Go**: Follow Go 1.25+ idioms (e.g., `any` instead of `interface{}`, `slices` and `maps` packages, `t.Context()` in tests).
- **Error Handling**: 
    - Use `internalError(c, err)` for unexpected errors to return a 500 status.
    - Use specialized helpers like `notFound(c)`, `validationError(c, details)`, and `conflict(c, message)` for client errors.
    - Always log errors on the server side before returning them to the client.
- **Dependency Injection**: The `Server` struct in `backend/api/api.go` holds dependencies like the database. Handlers are methods on this struct.
- **Validation**: Perform input validation in the handlers using helpers from `utils.go`. Do not rely solely on database constraints.
- **Contract Compliance**: Handlers must strictly follow the `ServerInterface` generated from the OpenAPI spec. Use `FromX` conversion functions in `utils.go` to map database models to API models.

## 2. Frontend (Vite, Svelte, TypeScript, openapi-typescript)

The frontend is a Svelte application built with Vite and TypeScript.

### Architecture
- **Type Safety**: TypeScript types are generated from the OpenAPI spec using `openapi-typescript`.
- **Generated Code**:
    - API type definitions are generated into `frontend/src/generated/api.d.ts`. This file should **never** be edited manually.
- **Handwritten Code**:
    - Svelte components in `frontend/src/lib/`.
    - Routes and pages in `frontend/src/routes/`.
    - API clients and fetch wrappers in `frontend/src/api/` (using the generated types).

### Guidelines
- **Strict Typing**: Use the generated `components["schemas"]` from `api.d.ts` for all data models coming from or going to the backend.

## 3. Persistence Layer (PostgreSQL, sqlc)

The project uses PostgreSQL for data storage, with `sqlc` for generating type-safe Go code from SQL.

### Architecture
- **Schema**: Defined in `backend/db/schema.sql`.
- **Queries**: Defined in `backend/db/query.sql`.
- **Views**: Use views (prefixed with `vw_`) to encapsulate complex joins and logic (e.g., soft-deletion filtering).
- **Generated Code**: `sqlc` generates `db.go`, `models.go`, and `query.sql.go` in `backend/db/`.

### Guidelines
- **Soft Deletion Policy**: 
    - **No Hard Deletes**: The system does not use hard deletions (`DELETE` statements). All data must be preserved for historical and audit purposes.
    - **Deleted Flag**: Tables that require deletion functionality (like `games` and `patrons`) must include a `deleted` boolean flag (defaulting to `FALSE`).
    - **Filtering**: Standard lookups and listings must filter out records where `deleted` is `TRUE`. Use database views (prefixed with `vw_`) to automatically handle this filtering to ensure consistency across the application.
- **Naming Conventions**: 
    - Use `snake_case` for table and column names in SQL.
    - Use descriptive names for `sqlc` queries (e.g., `-- name: GetGame :one`).
- **Indexes**: Ensure appropriate indexes are created for columns used in filters and joins (e.g., `idx_game_titles` on `sanitized_title`).
